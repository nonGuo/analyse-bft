# 📝 SQL加工脚本解析工具 - 需求规格说明书

## 1. 背景与目标

在数据仓库日常开发与运维中，数据加工脚本（SQL）的逻辑往往被黑盒化。为了提升数据治理能力、配合数仓平台的配置化和自动化测试，需要一款能够精准解析DWS环境下SQL脚本的工具，自动提取表级与字段级血缘，并生成标准化的Mapping文档和数据流图，以降低人工梳理的成本与错误率。

## 2. 功能需求

### 2.1 脚本预处理 (Pre-processing)

* **注释过滤：** 能够精准识别并剥离单行注释 (`--`) 和多行注释 (`/* ... */`)，但需保留可能存在的DWS执行计划Hint (`/*+ ... */`)。
* **变量替换：** 支持对数仓调度工具中常见的占位符变量（如 `${TX_DATE}`, `$bdp.system.bizdate` 等）进行识别和模拟值替换，防止破坏SQL语法树。
* **多语句分割：** 能够按分号 `;` 将复杂的长文本拆分为独立的SQL执行单元（DDL, DML, DCL）。

### 2.2 核心语法解析 (SQL Parsing)

* **目标语句识别：** 重点解析 `CREATE TABLE AS` (CTAS), `INSERT INTO ... SELECT`, `MERGE INTO`, 以及 `UPDATE` 语句。
* **CTE与子查询支持：** 必须支持深层嵌套的 `WITH ... AS (...)` 公用表表达式和内联子查询（Inline Views）的解析。
* **DWS特有语法兼容：** 忽略或兼容解析表属性定义（如 `WITH (ORIENTATION = COLUMN)`, `DISTRIBUTE BY HASH`, `PARTITION BY` 等），确保核心数据流不受影响。

### 2.3 血缘与逻辑提取 (Lineage & Logic Extraction)

* **表级血缘提取：** 准确提取目标表（Target）与所有依赖的源表（Source），理清依赖层级。
* **字段级血缘提取：** * 明确“目标字段 <- 来源表.来源字段”的映射关系。
* **加工逻辑捕获：** 提取字段级转换逻辑（如 `CASE WHEN`, `COALESCE`, 聚合函数等），并保留原始SQL片段。


* **别名追踪：** 能够精准绑定表别名和字段别名，解决隐式引用的问题。

### 2.4 产物生成 (Output Generation)

* **Mapping文档导出：** 生成结构化的Excel/CSV文件，包含字段：目标表、目标字段、数据类型、源表、源字段、转换规则。
* **数据流图生成：** 生成可视化的DAG（有向无环图），支持HTML交互式展示或静态PNG图片导出。

## 3. 非功能需求

* **元数据依赖感知：** 针对 `SELECT *` 或未指定前缀的字段，系统需具备元数据注入接口（通过连接DWS查询系统表或读取DDL缓存）以完成字段名推断。
* **高容错性：** 遇到部分无法识别的非核心专有语法时，应降级处理并记录Warning告警，而非直接导致程序崩溃（Crash）。
* **跨平台：** 纯Python实现，需完全兼容Windows与Ubuntu Linux开发/部署环境。

# 🏗️ SQL加工脚本解析工具 - 详细设计说明书

## 1. 技术选型

* **核心解析引擎：`sqlglot**`
* *原因：* 相比于基于正则的脆弱提取或过于严格的 `pglast`，`sqlglot` 纯Python编写，生成抽象语法树 (AST)，且原生支持自定义Dialect（方言）。我们可以通过继承它的 `Postgres` 方言，针对 GaussDB 8.2.1 注入特有词法规则。


* **图拓扑与数据流图：`networkx` + `pyecharts` (或 `Graphviz`)**
* *原因：* `networkx` 负责在内存中构建和遍历表/字段的DAG结构；`pyecharts` 用于生成交互式的数据流向网页，便于集成到数仓配置化平台中。


* **Mapping文档生成：`pandas` + `openpyxl**`
* *原因：* 字典数据直接转 DataFrame，一键输出美化版的 Excel。



## 2. 系统架构设计

系统整体分为五个核心模块（Pipeline 流水线架构）：

1. **Reader & Preprocessor (读取与预处理):** 清理文本，处理调度变量。
2. **Lexer & Parser (词法与语法分析):** 利用配置了DWS方言的 `sqlglot` 生成 AST。
3. **Scope Analyzer (作用域与血缘分析器):** 遍历 AST，进行别名绑定和作用域推导。
4. **Lineage Resolver (血缘消解):** 结合元数据，解决 `SELECT *` 和缺失别名的字段映射。
5. **Exporter (产物导出):** 将内存中的血缘对象转化为 Excel 和 Flowchart。

## 3. 核心模块详细设计

### 3.1 预处理模块设计

针对DWS的复杂环境，第一步是"净化"SQL。

```python
import re

def preprocess_sql(sql: str) -> str:
    # 1. 替换调度系统变量为假值 (防止语法错误)
    sql = re.sub(r'\$\{.*?\}', "'2026-01-01'", sql) 
    
    # 2. 剥离掉DWS特有的、不需要进入语法树的物理层DDL后缀 (降维打击)
    # 例如：WITH (ORIENTATION = COLUMN) DISTRIBUTE BY HASH(id);
    sql = re.sub(r'DISTRIBUTE\s+BY\s+HASH\s*\(.*?\)', '', sql, flags=re.IGNORECASE)
    sql = re.sub(r'WITH\s*\(\s*ORIENTATION\s*=\s*(COLUMN|ROW)\s*\)', '', sql, flags=re.IGNORECASE)
    
    return sql

```

### 3.2 自定义 DWS 方言设计 (基于 sqlglot)

为了让 `sqlglot` 看懂 GaussDB 的特性，我们需要定义一个轻量级的 Dialect。

```python
from sqlglot import dialects, exp

class DWSDialect(dialects.Postgres):
    class Parser(dialects.Postgres.Parser):
        # 在这里扩展DWS特有的关键字或函数
        FUNCTIONS = {
            **dialects.Postgres.Parser.FUNCTIONS,
            "NVL": exp.Coalesce.from_arg_list, # GaussDB常用NVL，等价映射到Coalesce
            "DECODE": exp.Case.from_arg_list,  # 转换Oracle风格的DECODE
        }
        
    class Generator(dialects.Postgres.Generator):
        pass

```

### 3.3 增加AI 协作模式

使用 sqlglot 作为核心，利用 AI 作为“智能外挂”是最佳路径

当遇到极其不规范、sqlglot 无论如何都解析不了的历史 GaussDB 脚本时，先调用大模型（配置特定 Prompt），让 AI 将其翻译/改写为标准的 PostgreSQL 语句（例如剥离特有 Hint、转换专有函数），然后再交给 sqlglot 提取结构。

### 3.4 作用域与血缘分析算法 (核心难点)

要从AST中提取Mapping，我们采用**自顶向下遍历，自底向上冒泡**的策略：

1. **捕获最外层：** 识别最外层的动作是 `INSERT` 还是 `CREATE`，从而锁定 **Target Table**。
2. **解析 CTE (WITH 语句)：**
* 将每个 CTE 视为一个临时的“虚拟表”，记录在当前解析上下文 (Context) 的注册表中。
* 对每个 CTE 执行递归解析，明确这个虚拟表的输出字段是由哪些底层物理表提供的。


3. **解析 SELECT 投影层：**
* 遍历 AST 的 `exp.Select` 节点。
* 针对每一个 `exp.Alias` (如 `A.col1 AS new_col`)，提取：
* **目标字段名：** `new_col`
* **转换逻辑 (Expression)：** 将 `A.col1` 的子树转化为字符串文本。
* **来源引用：** 识别 `A`，并去 `FROM/JOIN` 列表中查找 `A` 对应的真实表名或虚拟表名。




4. **处理连表操作 (JOINs)：** 记录 `ON` 条件，虽然不需要体现在最终的字段映射上，但对于生成准确的表级依赖图至关重要。

### 3.5 解决 `SELECT *` 的兜底设计 (Schema Fetcher)

在数仓自动化解析中，`SELECT *` 是一场灾难。工具必须设计一个 `MetadataProvider` 接口：

* **输入：** 表名 (如 `dim_user_info`)。
* **输出：** 该表的完整字段列表。
* **实现：** 工具在解析遇到 `exp.Star` 节点时，自动通过 `psycopg2` 连接 DWS 的 `pg_attribute` 或查询本地的数据字典缓存，展开这个 `*`，将其替换为具体的列名列表，再继续执行血缘分析。

### 3.6 产物导出数据结构

设计一套通用的内存数据模型，解耦解析器与生成器：

```python
from dataclasses import dataclass

@dataclass
class ColumnMapping:
    target_table: str
    target_column: str
    source_tables: list[str]  # 可能多个来源 (例如 a.id + b.id)
    source_columns: list[str]
    transformation_rule: str  # 原始SQL的片段，如 "COALESCE(a.amt, 0) * 100"

@dataclass
class LineageResult:
    target_table: str
    mappings: list[ColumnMapping]
    dependencies: list[tuple[str, str]]  # list of (source_table, target_table)

```

**数据流图生成逻辑：**
读取 `LineageResult.dependencies`，传入 `networkx.DiGraph`。

```python
import networkx as nx

def generate_dag(lineage: LineageResult):
    G = nx.DiGraph()
    for src, tgt in lineage.dependencies:
        G.add_edge(src, tgt)
    # 后续可对接 pyecharts 的 Graph 模块，或者直接将 G 转为 dot 文件交由 Graphviz 渲染成图

```

## 4. 实施与推进建议

1. **阶段一（表级验证）：** 优先使用 `sqlglot` 提取 CTE 和 FROM/JOIN 的表名依赖，跑通 `networkx` 的数据流图。这一步最容易实现，且立刻就能产出宏观架构图价值。
2. **阶段二（一对一字段级）：** 处理简单的 `INSERT INTO ... SELECT a, b FROM ...` 语句，跑通 Mapping Excel 导出。
3. **阶段三（函数与多对一）：** 攻克 `CASE WHEN`、嵌套子查询等复杂场景，并接入 DWS 真实表的 DDL 元数据解决跨表同名字段和 `SELECT *` 展开问题。
